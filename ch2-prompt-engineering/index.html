    Ch2 Prompt Engineering：把玄學變工程¶

    本章目標¶

    讀完本章你能：
    (1) 系統性地寫出穩定的 prompt，而不是憑感覺調；
    (2) 建立 prompt 的工程化管理流程（版控、測試、回滾）；
    (3) 說清楚 prompt 寫完後怎麼判斷好不好。

    2.1 核心心法：Prompt 是規格書，不是咒語¶

    想像你把工作外包給一個能力極強、但完全按字面理解、且每次都是第一天上班的工程師。你寫給他的工作說明書，就是 Prompt 。

    這個比喻決定了所有原則：

    • AI 不知道你公司的名詞——該定義的名詞要定義
    • AI 不知道「你懂我意思吧」——模糊的要求得到模糊的結果
    • AI 每次都忘記上次教過的——規則要每次都在說明書裡
    • 給 AI 看範例比給他形容詞有效——「像這樣做」勝過「做得專業一點」

    寫 prompt 的功力，本質上是寫規格的功力。

    2.2 Prompt 的標準解剖¶

    一個生產等級的 prompt 通常包含這些部分（順序也大致如此，配合 prompt caching 固定在前）：

    [系統提示 System Prompt]
1. 角色與任務：「你是保險理賠文件的分類器」
2. 規則與邊界：「只能輸出以下五類之一；資訊不足時輸出 UNKNOWN，禁止猜測」
3. 輸出格式：明確的 JSON schema
4. 範例（few-shot）：2–5 組 輸入→正確輸出

[使用者輸入 User Message]
5. 本次要處理的實際內容

    幾個關鍵細節：

    • 規則用「要做什麼」優於「不要做什麼」。「只輸出 JSON」比「不要輸出多餘文字」穩定。
    • 給出口：明確告訴它資訊不足時怎麼辦（輸出 UNKNOWN／說「文件中沒有提到」）。不給出口，它就會硬掰——幻覺很多時候是 prompt 沒給「我不知道」的選項。
    • Few-shot 是最便宜的性能提升：格式不穩、分類飄移，先加範例再說。範例要涵蓋邊界情況（一個正常件、一個模糊件、一個該回 UNKNOWN 的件）。

    2.3 Chain of Thought：先想再答¶

    要求模型 「先逐步推理，再給最終答案」（CoT），在複雜任務上準確率明顯提升。原理用接龍比喻很好懂：直接要答案，等於要它「一步接出正確結論」；讓它先推理，等於讓它把中間步驟寫在桌面上，後面的接龍有所依據。

    代價：更多 output token（錢）與延遲。所以： 簡單任務（格式轉換、明確分類）不用 CoT；需要判斷、多步推導的才用。 現代的推理型模型（reasoning models）是把這件事內建在模型層——使用時就不必再在 prompt 裡手動要求。

    2.4 Structured Output：永遠不要相信模型會乖¶

    企業整合的鐵律：模型輸出要進系統，就必須是可驗證的結構化格式。標準做法三件套：

    def extract(document: str, retries: int = 2) -> Claim:
    for attempt in range(retries + 1):
        raw = llm(SYSTEM_PROMPT, document)      # 1. prompt 要求 JSON（給 schema 與範例）
        try:
            return Claim.model_validate_json(raw)  # 2. 程式端用 schema 驗證（pydantic）
        except ValidationError as e:
            document = f"{document}\n\n上次輸出格式錯誤：{e}，請修正"  # 3. 帶著錯誤重試
    raise ExtractFailed(document)               # 重試耗盡 → 進人工佇列，絕不讓髒資料下行

    三個層次缺一不可：要求格式 → 驗證 → 失敗重試與兜底。多數 API 也提供原生的 structured output / JSON mode，能用就用——但程式端驗證仍然不能省，這是信任邊界。

    2.5 Prompt 的工程化管理：Prompt is Code¶

    Prompt 直接決定系統行為——所以它就是程式碼，要用管理程式碼的方式管理它：

    程式碼的做法 Prompt 的對應
    版本控制 Prompt 進 git，不散落在程式字串或誰的記事本裡
    測試 每次改動跑 eval set 回歸（第六章）
    Code review Prompt 改動走 PR 流程
    部署與回滾 Prompt 有版本號，線上問題能一鍵回舊版
    A/B 測試 新舊 prompt 分流比較線上指標

    prompt 寫完怎麼知道好不好：「用 eval set 回歸測試，不是肉眼看兩個例子」。改一個字可能讓某類輸入變好、另一類變壞，肉眼根本看不出來，只有成套的測試集能看出來。

    2.6 實務調試流程（prompt 不穩時的 debug 順序)¶

    1. 先看模型實際收到什麼——歷史被裁掉？變數沒代入？八成的「prompt 問題」其實是組裝問題。
    2. 縮小重現——找出最小的會錯的輸入。
    3. 檢查是否缺出口——它亂掰是不是因為你沒允許它說不知道。
    4. 加範例——針對錯的型態加一個 few-shot 範例。
    5. 拆步驟——一個 prompt 做五件事就會五件都平庸；拆成 pipeline，每步單純。
    6. 每次改動跑回歸——不要「感覺變好了」。

    常見誤解¶

    1. 「Prompt engineering 就是話術技巧」——生產環境的 prompt 工程是：規格撰寫＋格式驗證＋測試回歸＋版本管理。話術只佔一成。
    2. 「寫一次調到好就不用動了」——模型會升級、輸入分布會漂移。沒有回歸測試的 prompt 是定時炸彈。
    3. 「規則寫愈多愈保險」——規則互相衝突、或把重點淹沒在中段（lost in the middle），反而更不穩。規則要少而明確，複雜邏輯拆到程式端或多步 pipeline。
    4. 「模型輸出錯格式是模型爛」——沒做驗證與重試就上線，是工程師的鍋。

    自我檢測¶

    1. 客戶的分類 prompt 準確率不穩，你的 debug 順序是什麼？（至少五步）
    2. 什麼是 few-shot？範例應該怎麼挑？
    3. 寫出 structured output 三件套，並解釋為什麼有原生 JSON mode 還是要程式端驗證。
    4. 「prompt 寫完怎麼知道好不好？」——用完整、有條理的答案回答。
    5. 為什麼 prompt 要「給出口」？跟幻覺有什麼關係？

    參考答案要點¶

    1. 看實際輸入 → 最小重現 → 檢查出口 → 加範例 → 拆步驟 → 回歸測試。
2. Few-shot＝在 prompt 裡直接放幾組「輸入→正確輸出」範例，示範給模型看。挑範例別全放簡單的，要湊齊三種：一個正常件、一個模糊的邊界件、一個該回 UNKNOWN 的拒答件。
3. 要求格式→schema 驗證→帶錯重試→兜底人工；驗證是信任邊界，模型輸出永遠可能出格。
4. Eval set 回歸，不是肉眼；改動可能此好彼壞，只有測試集看得出來。
5. 不給「我不知道」的選項，模型被迫接龍硬掰；出口是最便宜的防幻覺手段之一。
