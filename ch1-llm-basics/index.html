    Ch1 LLM 是怎麼運作的¶

    本章目標¶

    讀完本章你能：
    (1) 用白話向客戶解釋 LLM 的運作原理與天生限制；
    (2) 看懂 API 帳單並知道成本怎麼壓；
    (3) 說清楚 LLM 有哪些天生限制，以及對應的工程對策。

    1.1 一個比喻起手：超強的接龍機器¶

    LLM 本質上是一台「接龍機器」：給它一段文字，它預測「下一個字最可能是什麼」，寫出來，再預測下一個，一路接下去。它讀過網路上巨量的文字，所以接得非常好——好到能寫程式、能推理、能對話。

    這個比喻立刻解釋了三件事：

    1. 它輸出的是「最可能的下文」，不是「查證過的事實」——所以會有幻覺（hallucination）：一本正經地說錯話。它不是在騙你，它是在接龍。
    2. 它沒有「資料庫」——知識壓在模型參數裡，模糊、不可更新、無法引用出處。想讓它用你公司的資料，要自己餵給它（第三章 RAG 就是在做這件事）。
    3. 它每次都是重新接龍——模型本身沒有記憶。「它記得我們的對話」是假象：是應用程式把歷史對話每次重新塞給它。

    對客戶講的版本：「AI 像一個讀過全世界文件、記憶力驚人但從不查證的顧問。我們的工程工作，就是幫他配一個資料庫、一套查證流程、和一個權限系統。」

    1.2 Token：模型世界的計量單位¶

    模型不是一個字一個字處理文字，而是一個 token 一個 token。Token 是「常見字串片段」：英文大約 4 個字元一個 token，中文大約一個字 1–2 個 token。

    為什麼你必須在意 token？因為 LLM 世界的三樣東西都用它計量：

    東西 以 token 計 實務意義
    成本 input / output 分開計價 output 通常貴好幾倍。廢話輸出=燒錢
    容量 context window 上限 塞不下就要裁切或摘要
    速度 逐 token 生成 要求短輸出=更快回覆

    1.3 Context Window：模型的工作記憶¶

    Context window（上下文視窗） 是模型單次能「看見」的 token 總量上限——包含你給它的所有東西（系統提示、對話歷史、文件）加上它要生成的回覆。

    把它想成工作桌面：桌面很大（現代模型十萬到百萬 token），但終究有限。桌面放不下的東西，模型就是看不見——不管那份文件多重要。

    三個實務推論：

    1. 長對話會「失憶」：歷史超過視窗，早期內容被裁掉。解法：摘要壓縮、或把重點存外部再檢索回來。
    2. 「把全公司文件都丟進去」行不通：量太大、太貴、而且——
    3. Lost in the middle：就算塞得下，模型對視窗「中段」內容的注意力較弱，重點放開頭與結尾效果最好。塞愈多雜訊，關鍵資訊愈容易被稀釋。「塞更多」不等於「答更好」，給對的少量內容勝過給全部——這是 RAG 的核心哲學，第三章詳談。
    4. 適當的在完成階段性任務的時候就 /clear ， 反而會常常讓接下來的事情進行更順利也更快完成。

    1.4 Temperature：控制隨機性的旋鈕¶

    模型預測下一個 token 時，其實算出的是一個機率分布（「的」40%、「了」20%、「呢」5%⋯）。Temperature 控制它怎麼從分布裡挑：

    • 低（0–0.3）：幾乎總是挑最高機率 → 輸出穩定、可重現。用於：分類、抽取、程式碼、企業流程——FDE 日常 95% 的場景。
    • 高（0.7 以上）：允許挑機率較低的選項 → 多樣、有創意、但不穩定。用於：文案發想、腦力激盪。

    常見誤解：temperature 0 不代表「更聰明」或「絕對不會錯」，只代表「同樣輸入給同樣輸出」。錯的答案也會穩定地錯。

    1.5 Streaming 與延遲¶

    模型逐 token 生成，所以可以邊生成邊回傳（streaming）。使用者體驗的關鍵指標是 TTFT（time to first token，首字延遲）——使用者看到第一個字的等待時間。總生成時間 10 秒但 TTFT 0.5 秒的體驗，遠好於總時間 5 秒但要黑屏等 5 秒。

    企業場景的延遲優化清單：更小的模型 → streaming → prompt 精簡 → 快取 → 平行化前置步驟（如檢索）。

    1.6 成本模型與 Prompt Caching¶

    API 計價：input token 與 output token 分開算，output 貴數倍。

    Prompt caching（提示快取）：如果每次請求的開頭都一樣（同一份系統提示、同一批文件），供應商可以把這段「已讀過的前綴」快取起來，下次不用重新處理——成本大幅下降、速度變快。設計 prompt 時把「固定不變的放前面、每次變動的放後面」，就是為了吃到快取。

    成本優化手段總表（第九章會再展開）：

    1. 模型分級——簡單任務用便宜小模型，難任務才用旗艦
    2. Prompt caching——固定前綴
    3. 控制輸出長度——要 JSON 就別讓它寫散文
    4. Batch API——不急的工作批次跑，通常半價
    5. 快取重複問題——同樣的問題別問第二次

    1.7 天生限制總整理¶

    被問「LLM 有什麼限制」，用這個結構答，每個限制都接一個對策：

    限制 原因 工程對策
    幻覺 接龍的機率本質，不查證 RAG＋引用出處＋evals(AI 的評估機制)＋人審關卡
    知識過期 訓練有截止日 RAG／工具（搜尋、查 DB）
    不會真的計算 語言模型不是計算機 給它計算工具（tool use）
    視窗有限＋中段失焦 架構限制 檢索精選內容，不硬塞
    輸出不穩定 機率生成 低 temperature＋structured output＋驗證重試
    無法區分指令與資料 一切都是文字 權限層防禦（第八章）

    一句話總結：「LLM 的每個限制都有成熟的工程對策——所以企業導入 AI 的瓶頸從來不是模型不夠聰明，而是周邊工程沒做齊。這正是 FDE 存在的原因。」

    常見誤解¶

    1. 「模型會上網查資料」——不會，除非你明確給它搜尋工具。原生的它只有訓練時的知識。(編者註:加入 Skill， 例如 claude-in-chrome， 就會上網查資料， 但不一定每次都會， 如果需要最好在 Prompt 內加入)
    2. 「它記得我們上次聊什麼」——不記得。是應用程式重新餵歷史。這也意味著：你餵什麼它就知道什麼，你沒餵它就不知道——除錯 AI 應用時第一件事永遠是「看它實際收到了什麼」。(編者註:即便是同一個 Session ， 也可能因為 Context 太長混淆造成記憶出現問題)
    3. 「它講得很有自信，應該是對的」——自信語氣是文風，不是可信度指標。幻覺最危險的地方就是流暢自信， 所謂的一本正經講幹話的 AI 就是這個意思。
    4. 「換更大的模型就能解決」——大模型降低幻覺率但不歸零；沒有 RAG 與 evals 的系統，換什麼模型都不可靠。

    自我檢測¶

    1. 用一個比喻向完全不懂技術的客戶解釋「為什麼 AI 會一本正經地講錯話」，並說出你的工程對策。
    2. 客戶抱怨「AI 回覆太慢」——列出你的延遲診斷與優化順序。
    3. 什麼是 prompt caching？設計 prompt 時要怎麼配合它？
    4. 為什麼「把所有文件都塞進 context」是壞主意？（要講到成本、視窗、lost in the middle 三點）
    5. Temperature 設 0 之後，輸出就保證正確嗎？為什麼？

    參考答案要點¶

    1. 接龍機器/從不查證的顧問比喻；對策=RAG＋引用＋evals。
2. 先量測（TTFT vs 總時長）→ streaming → 模型分級 → prompt 精簡 → 快取 → 平行化檢索。
3. 固定前綴的處理結果被快取；固定內容放前、變動放後。
4. 貴（token 計費）、放不下（視窗上限）、放下了也失焦（lost in the middle）；給對的少量內容勝過給全部。
5. 不保證。只保證可重現——同輸入同輸出，錯也穩定地錯。正確性靠 evals 驗證。
