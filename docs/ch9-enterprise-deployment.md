# Ch9 企業部署：從 Demo 到 Production

## 本章目標

讀完你能：(1) 講清楚 demo 與 production 之間差了哪些工程；(2) 回答「託管 API vs 私有部署」的顧問級比較；(3) 解釋 SSO/RBAC/審計三件套；(4) 面對台灣企業（金融、製造）的法遵語境不怯場。這章是 FDE 與一般 AI 工程師的分水嶺。

---

## 9.1 Demo 到 Production 的鴻溝

週五下午你給客戶 demo，效果驚豔，CEO 當場拍板。然後呢？

Demo 的世界：一個使用者、乾淨的樣本資料、你的筆電、出錯重來就好。
Production 的世界：三千個員工、二十年的髒資料、客戶的內網、出錯上新聞。

中間差的工程，就是 FDE 的工作清單：

1. **身分**：三千人怎麼登入？（SSO）
2. **權限**：業務能看的，工讀生能看嗎？（RBAC＋檢索層過濾）
3. **審計**：金管會來查，能不能重建「誰在何時問了什麼、AI 答了什麼」？
4. **可靠性**：API 掛了怎麼辦？重試、降級、監控、告警
5. **品質**：上線後怎麼知道它還好著？（evals＋線上訊號，Ch6）
6. **變更管理**：改 prompt 要走什麼流程？（版控＋回歸＋審核）

> 面試金句：「demo 證明的是『模型做得到』，production 要解決的是『組織敢不敢用』——中間全部是工程與治理，這正是 FDE 的價值所在。」

## 9.2 託管 API vs 私有部署（顧問級必答題）

客戶問：「資料敏感，我們是不是要自己架模型？」——用四象限回答：

| 象限 | 託管 API（OpenAI/Anthropic 等） | 自建開源模型（Llama 等） |
|---|---|---|
| **能力** | 前沿模型，持續免費升級 | 開源追趕中，窄任務可 fine-tune 補 |
| **資安/法遵** | 資料出企業網（有企業條款保障） | 資料完全不出內網 |
| **成本** | 按量計費，起步零基建 | GPU 基建＋維運團隊，量大才攤得平 |
| **維運** | 零（人家管） | 全自扛：部署、擴縮、更新、安全補丁 |

**關鍵的中間解（台灣企業的甜蜜點，一定要講）**：**雲上私有端點**——Azure OpenAI、GCP Vertex AI 上的模型服務。前沿模型的能力＋資料不離開企業的雲租戶環境＋符合多數法遵要求，不用自扛 GPU。**多數「我們要自建」的需求，聽完這個選項就轉向了。**

誠實提醒客戶自建的隱形成本：不只是買 GPU——是養一個模型維運團隊（推理優化、版本管理、安全更新），而且模型能力落後前沿一個世代。「先用私有端點上線驗證價值，真有理由再評估自建」是負責任的路徑。

## 9.3 企業整合三件套：SSO、RBAC、審計

### SSO（單一登入）

員工用公司帳號登入所有系統，背後協定是 **SAML** 或 **OIDC**（新系統主流）。白話：你的 AI 應用不自己管密碼，而是「信任公司的身分中心（Okta、Azure AD）開出的通行證」。FDE 要會的是：接 OIDC 流程、把通行證裡的身分與群組資訊接進應用。

### RBAC（角色權限）

身分進來了，接著是「這個人能看什麼」。AI 系統的特殊之處（重要）：**權限要一路下沉到檢索層**——業務部的人問問題，檢索範圍就只有業務部的文件（Ch3/Ch8 講過：靠 metadata 過濾，不靠 prompt）。從 SSO 的群組映射到文件的權限標籤，這條線是企業 RAG 最容易做錯的地方。

### 審計（Audit Trail）

誰、何時、問了什麼、系統檢索了什麼、AI 答了什麼、人有沒有覆核——全程留痕。這不是 nice-to-have：金融業是法遵硬要求，也是出事後自證清白的唯一辦法。（日誌本身含敏感資料，遮罩與存取控管比照個資。）

## 9.4 可靠性工程：AI 系統的 SRE 基本功

外部 LLM API 是你架構裡「最不可靠的依賴」，按對待任何脆弱外部依賴的方式對待它：

- **重試**：exponential backoff＋jitter（coding 題 C2 就是它）；區分可重試（429/超時）與不可重試（400）
- **降級**：主模型掛了 → 備用模型／快取答案／「目前無法回答」的優雅失敗——**永遠不要讓 AI 故障變成業務停擺**
- **限流與配額**：保護自己（成本）也保護上游（429）
- **監控四金標**：延遲（TTFT/總時長）、流量、錯誤率、飽和度（token 配額用量）——外加 AI 特有的：eval 分數趨勢、幻覺回報率、轉人工率

## 9.5 變更管理：企業節奏

企業有凍結期、UAT、上線窗口、回滾要求。你的 AI 系統要能配合：

- Prompt/模型版本化，任何改動可回滾（Ch2）
- 改動附 eval 回歸報告才能過審（Ch6）——「這次改動在 500 個測試案例上的分數變化」就是你的 UAT 證據
- 灰度發布：新 prompt 先 5% 流量

> 「隨時能改」在新創是優點，在企業是風險。把「快速迭代」包裝成「每次迭代都有回歸證據的可控變更」，企業才敢讓你快。

## 9.6 台灣語境速查

- **金融**：金管會監理。資料在地、委外雲端服務有專門規範、稽核軌跡硬要求——你的台新經驗直接對話
- **個資法**：個人資料進 AI 系統的告知與目的限制；日誌遮罩
- **製造業**：機密是製程參數與良率資料（商業機密級），常見要求 on-prem 或不出廠區；網路隔離環境部署（AIFT 的 JD 就在講這個）
- 共同心理：不是抗拒 AI，是怕失控。你的答案永遠是治理框架（權限/審計/人審），不是「相信模型」

## 常見誤解

1. **「Demo 成功＝快上線了」**——demo 是起點；三件套＋可靠性＋變更管理才是工期主體。報時程時把這些算進去，否則就是把自己埋進去。
2. **「資料敏感＝必須自建模型」**——先講雲上私有端點；自建是最後選項且要算全成本。
3. **「權限用 prompt 控就好」**——第三次強調：檢索層過濾。這在面試是紅線題，答錯直接出局。
4. **「AI 系統的監控就是看 API 有沒有活著」**——還要監控品質（eval 趨勢、轉人工率）。系統活著但答案劣化，是更陰險的故障。

## 自我檢測

1. 客戶 demo 後問「多久能全公司上線？」——列出 demo 到 production 的工程清單（六項）。

3. 白話解釋 SSO/RBAC/審計三件套，並指出 AI 系統在 RBAC 上的特殊點。
4. 設計 AI 系統的可靠性：重試、降級、監控各要點。
5. 金融客戶的稽核部門問「AI 的決策能被稽核嗎？」——你的回答（用上台新經驗）。
