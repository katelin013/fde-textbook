# Ch7 技術選型：Prompting vs RAG vs Fine-tune

## 本章目標

讀完你能：(1) 面對任何需求快速判斷該用哪招；(2) 向客戶解釋為什麼「不用最貴的那招」；(3) 說清楚什麼時候該 fine-tune、什麼時候該用 RAG、什麼時候純 prompt 就夠。

---

## 7.1 三招是什麼（一句話版）

| 招式 | 一句話 | 改變的是什麼 |
|---|---|---|
| **Prompting** | 把要求和範例寫進說明書 | 模型的**行為指示** |
| **RAG** | 考試時翻參考書 | 模型可用的**知識** |
| **Fine-tune** | 送模型去受訓，改變它的本能 | 模型的**內建傾向**（風格/格式/領域行為） |

## 7.2 黃金決策順序：由便宜到貴

**先 prompt → 不夠再 RAG → 最後才考慮 fine-tune。**

理由不只是省錢——是**迭代速度與可維護性**：

- Prompt 改一行，一分鐘後生效，回歸測試跑完就能上
- RAG 更新知識＝更新文件庫，不動模型
- Fine-tune 每次調整都要重新準備資料、訓練、評估、部署——迭代以「週」計

> 一句話總結：「**Fine-tune 教模型『怎麼說』，RAG 給模型『說什麼』。** 企業要的多半是後者，而且要能引用出處——所以我的預設路徑是 prompt → RAG，fine-tune 是有明確理由才動用的最後手段。」

## 7.3 決策因素逐一拆解

**看知識還是行為？**
- 缺**知識**（公司文件、產品資料、最新資訊）→ RAG。Fine-tune 灌知識效果差又無法引用出處，還會過期。
- 缺**行為**（固定的輸出格式、領域語氣、特殊術語習慣）→ 先試 prompt＋few-shot；範例給了還是學不像、且需求量大到在乎單次成本 → 才考慮 fine-tune。

**需要出處引用嗎？**
- 需要（企業幾乎都需要）→ RAG 必選。Fine-tune 做不到「這句話來自哪份文件」。

**知識多久更新一次？**
- 天天變 → RAG（改文件即生效）。Fine-tune 的知識凍結在訓練那天。

**有多少高品質範例？**
- Fine-tune 需要數百到數千筆高品質「輸入→理想輸出」。沒有這個量與品質，練出來的是災難。

**在乎單次延遲與成本嗎（規模大）？**
- Fine-tune 的正當理由之一：用小模型 fine-tune 後取代大模型做特定窄任務——量大時省錢省延遲。這是「蒸餾」思路。

## 7.4 Fine-tune 的真實成本（客戶只看到訓練費，你要講全）

1. 資料準備：數百～數千筆標註（最貴的其實是這個）
2. 訓練與調參：多輪實驗
3. **評估**：要有自己的 eval set 證明沒練壞（練好 A 能力、退化 B 能力很常見）
4. 部署與維運：自己管模型版本
5. **每次更新重來一遍**：基礎模型升級了？你的 fine-tune 要重做

所以對客戶的標準說法：「fine-tune 不是一次性採購，是養一條訓練 pipeline。除非有明確且持續的理由，否則先用 prompt＋RAG 達到八成效果，把預算留給 eval 與整合。」

## 7.5 三個實戰情境

**情境一：法律事務所要 AI 查過往判例與所內備忘錄。**
→ RAG。知識型、常更新、必須引用出處。Fine-tune 完全不對症。

**情境二：客服中心要 AI 回覆風格完全符合品牌語氣，每天十萬則。**
→ 先 prompt＋few-shot 試（多半夠）；若語氣仍不穩＋量大在乎成本 → 用歷史優質回覆 fine-tune 小模型。知識部分（產品資料）仍配 RAG——**兩招常常是組合，不是單選**。

**情境三：製造業要 AI 讀懂自家縮寫黑話（「T3 站的 QN 單」）。**
→ 第一步竟然是 prompt：把術語表放進系統提示（或 RAG 檢索術語表）。詞彙量大到 prompt 放不下再談其他。**很多「我們需要 fine-tune」的需求，實際上是一份術語表就解決了。**

## 常見誤解

1. **「Fine-tune 是最高級的，所以效果最好」**——是最貴的，不是最對的。多數企業需求是知識型，RAG 對症。
2. **「RAG 跟 fine-tune 二選一」**——常常組合：fine-tune 管風格、RAG 管知識、prompt 管規則。
3. **「fine-tune 能讓模型記住我們公司的所有文件」**——效率差、無法引用、會過期。灌知識用 RAG。
4. **「prompt 太長就該 fine-tune」**——先看 prompt caching（固定前綴幾乎免費），再看 RAG（動態塞需要的），fine-tune 是第三順位。

## 自我檢測

1. 「什麼時候 fine-tune、什麼時候 RAG、什麼時候純 prompt？」——完整決策框架 3 分鐘作答。
2. 客戶堅持「我們要 fine-tune，聽說最厲害」——你怎麼回？（要有同理、有選項、有取捨）
3. 7.5 三個情境不看答案重answer一遍。
4. Fine-tune 的五項真實成本是什麼？

## 參考答案要點

    1. 順序（便宜→貴）＋因素（知識vs行為、引用、更新頻率、範例量、規模成本）＋金句。
    2. 先問需求本質（要知識還是要風格？）→ 展示 prompt/RAG 能達到的效果與時程 → fine-tune 的全成本 → 給階段性方案（先 RAG 上線，量化差距後再評估 fine-tune）。
    3-4. 見 7.4、7.5。
